## Root Cause Analysis - More Detailed

### What We Now Know

1. Korean strings are present in the built binary.
2. The system code page is Korean (`ACP=949`), and text conversion was attempted before calling the game text native.
3. English renders correctly through the current overlay path.
4. Korean game subtitles outside our overlay render correctly in the base game.
5. Our overlay specifically shows square replacement glyphs instead of readable Hangul.

### What That Means

This strongly suggests the active overlay text path is using a font atlas or debug UI font that does not include Hangul glyphs.

Current path:

- `VAR_STRING("LITERAL_STRING", ...)`
- `_BG_DISPLAY_TEXT`

This path appears valid for English/ASCII, but not sufficient for Korean display in this overlay context.

### Why This Is Important

That means tomorrow's work should not focus first on:

- string translation quality
- INI config changes
- zone radius tuning
- UTF-8 byte generation

Those are no longer the primary blockers.

The real blocker is:

- moving Korean display to a UI rendering route that supports Hangul glyphs
